 # AI DM 产品架构文档
 
 **版本**：v0.2（优化版）
 **日期**：2026-07-08
 **范围**：MVP（两个月交付周期），以 Call of Cthulhu 7th 单剧本为首次落地场景
 
 ---
 
 ## 一、产品定位与核心原则
 
 ### 1.1 一句话定位
 
 > **AI DM = 一个让朋友们在手机上扫码即玩、AI 代替真人 KP 主持跑团故事的 Web 应用。**
 
 产品形态本质上是一个 **智能多人聊天室**：类似微信群聊，但 AI 是群主兼主持人，负责推剧情、旁白描写、扮演 NPC、过检定、私发秘密线索。
 
 ### 1.2 核心设计原则
 
 | 原则 | 解释 |
 |---|---|
 | **零门槛上车** | 无需注册复杂账号、无需学规则书、无需地图 Token，扫码进房间直接开始 |
 | **多人优先** | 从架构第一天支持多玩家联机，信息隔离（每人只看自己角色知道的秘密） |
 | **混合架构** | 确定性规则引擎保证游戏不走样 + AI 生成保证沉浸感，不互相替代 |
 | **剧本即配置** | CoC 剧本以结构化数据定义（剧情节点、线索图、NPC 卡、怪物数据），引擎可加载不同剧本 |
 | **规则书知识库** | 规则书全文通过 RAG 索引，AI 运行时按需检索，减少硬编码依赖 |
 | **模组上传即玩** | 玩家上传模组 PDF/文档，AI 自动提取场景、NPC、线索，生成可加载的剧本包 |
 
 ### 1.3 目标用户
 
 1. **想跑团但没有 KP 的小团体**（核心用户，2-6 人）
 2. **线下聚会场景的破冰活动**（桌游吧、团建）
 3. **想体验 TRPG 但不敢入坑的新人**（AI 引导 + 自动规则）

 ## 二、AI 调用全景图（本次重点）
 
 这是本版本文档的核心更新——明确哪些地方调用 AI，哪些地方走确定性代码。
 
 ### 2.1 一句话概括
 
 > **规则执行交给代码，叙事和角色交给 AI，规则书检索做 AI 的"外挂大脑"。**
 
 ### 2.2 全景图
 
 ```
 ┌──────────────────────────────────────────────────────────────────┐
 │                        玩家输入                                    │
 │              "我想翻一下办公桌的抽屉"                               │
 └────────────────────────┬─────────────────────────────────────────┘
                          │
                          ▼
 ┌──────────────────────────────────────────────────────────────────┐
 │  🔵 AI 调用：Intent Parser                                       │
 │  输入: "我想翻一下办公桌的抽屉"                                   │
 │  输出: { action: "search", target: "desk_drawer" }              │
 │  兜底: 关键词匹配 + 场景可交互对象列表                            │
 └────────────────────────┬─────────────────────────────────────────┘
                          │
                          ▼
 ┌──────────────────────────────────────────────────────────────────┐
 │  🟢 确定性代码：Scene Controller                                 │
 │  - 检查"办公桌"在当前场景是否存在                                │
 │  - 确定需要过什么检定（Spot Hidden）                             │
 │  - 查找检定难度等级                                              │
 └────────────────────────┬─────────────────────────────────────────┘
                          │
                          ▼
 ┌──────────────────────────────────────────────────────────────────┐
 │  🟢 确定性代码：Rule Engine (Skill Check)                        │

 │  - 计算成功率: 技能值 65 vs 骰出 47 → 成功                      │

 │  - 更新状态: 标记线索 diary_fragment_1 为"已发现"                │
 └────────────────────────┬─────────────────────────────────────────┘
                          │
                          ▼
 ┌──────────────────────────────────────────────────────────────────┐
 │  🔵 AI 调用：Narration Generator                                │
 │  输入: 检定结果 + 场景上下文 + 线索内容                          │
 │  输出: "你在抽屉夹层里摸到一个泛黄的旧信封……"                     │
 └────────────────────────┬─────────────────────────────────────────┘
                          │
                          ▼
 ┌──────────────────────────────────────────────────────────────────┐
 │  🟢 确定性代码：Event Log + 消息分发                             │

 │  - 按 visibility 字段分发（public / whisper）                    │
 └──────────────────────────────────────────────────────────────────┘
 ```
 
 ### 2.3 AI 调用 vs 确定性代码 完整划分
 
 #### 🟢 确定性代码（不调 AI，全硬编码逻辑）
 
 | 模块 | 具体内容 |
 |---|---|
 | **骰子引擎** | 掷骰、奖惩骰计算、结果判定 |
 | **技能检定结算** | 技能值 vs 骰值 → 成功/失败/大成功/大失败 |
 | **Sanity 计算** | 掉 san 值、暂时疯狂/不定疯狂判定 |
 | **战斗轮系统** | 先攻排序、行动选择、命中/伤害计算 |
 | **HP 管理** | 伤害扣除、治疗恢复、死亡判定 |
 | **场景状态机** | 场景进入条件、推出条件、线索解锁 |
 | **线索图导航** | 已发现线索 → 可进入的新场景 |
 | **房间管理** | 创建/加入/离开/超时销毁 |
 | **信息隔离** | 消息 visibility 校验、分发路由 |
 | **事件日志** | 不可变日志写入、查询 |
 
 #### 🔵 AI 调用（必须调 LLM）
 
 | 模块 | 具体内容 | 兜底策略 |
 |---|---|---|
 | **意图解析** (Intent Parser) | 玩家自然语言 → 结构化游戏动作 | 关键词匹配 + 场景可交互列表菜单 |
 | **旁白生成** (Narration) | 场景描写、检定结果叙事、氛围渲染 | 预设文本模板 |
 | **NPC 对话** (NPC Chat) | 扮演 NPC，角色互动对话 | 预设对话树 |
 | **规则查询** (Rule Query) | 玩家问"规则书里关于极端成功的规则是什么" | 直接返回规则书原文段落 |
 | **引导提示** (Guidance) | 玩家卡关时给出隐晦提示 | 预设 NPC 提示列表 |
 | **模组解析** (Module Parser) | 上传模组 → 提取场景/NPC/线索 | 人工导入 JSON 兜底 |
 | **复盘生成** (Replay Gen) | 游戏结束后生成叙事性复盘 | 按时间线排列事件摘要 |
 
 #### 🟡 可 AI 可确定性的模块（根据场景切换）
 
 | 模块 | AI 模式 | 确定性模式 |
 |---|---|---|
 | **角色车卡** | AI 根据模组风格推荐预设角色 | 手动填表 / 随机生成 |
 | **行动选项推荐** | AI 根据场景推荐 3 个可做的事 | 列出场景内所有可交互对象 |
 | **边缘规则裁决** | AI 查知识库解释罕见规则 | 返回"该规则暂未实现" |
 
 ### 2.4 AI 调用策略（成本与延迟控制）
 
 | 策略 | 说明 |
 |---|---|
 | **流式输出** | AI 生成旁白/NPC 对话时使用 SSE/Stream，减少用户等待感知 |
 | **占位缓冲** | AI 响应前，先显示"杰克翻找着抽屉……"这类过渡文本 |
 | **上下文预算** | 每次 AI 调用严格控制在 ~2500 token 以内，防止成本和延迟爆炸 |
 | **缓存复用** | 同一场景、同一 NPC 的首次生成结果可缓存，减少重复调用 |
 | **降级路径** | AI 超时/报错 → 自动切到预设文本或操作菜单，游戏不中断 |

 ## 三、规则书知识库 + 模组自动解析（新架构层）
 
 你提到"把规则书做成知识库，玩家上传模组，AI 自动生成需要的内容"——这个思路非常好，但它不是替代确定性代码的，而是**补充和增强**。下面是我对这个方案的具体设计。
 
 ### 3.1 三层架构：三个角色，各司其职
 
 ```
 ┌─────────────────────────────────────────────────────────────────┐
 │  ① 规则书知识库（RAG）                                          │
 │     角色: AI 的"外挂大脑"                                       │
 │     存什么: CoC 规则书全文、官方 FAQ、村规补充                   │
 │     谁用: AI Engine（运行时按需检索）                            │
 │     不用来: 执行规则（规则执行是代码的事）                       │
 ├─────────────────────────────────────────────────────────────────┤
 │  ② 硬编码规则引擎（Deterministic Code）                         │
 │     角色: 游戏的"物理引擎"                                      │
 │     做什么: 技能检定、san 计算、战斗结算                        │
 │     为什么不能交给 AI: 玩家骰出 01 大成功，AI 错误理解成失败    │
 │     就会毁掉一个关键剧情时刻，这种事情不允许发生                   │
 ├─────────────────────────────────────────────────────────────────┤
 │  ③ 模组解析器（AI Module Parser）                               │
 │     角色: 上传模组 → 自动生成可加载剧本包                       │
 │     做什么: 读模组文档 → 抽场景/NPC/线索/怪物 → 输出 JSON      │
 │     人工审核: 生成结果可以在 Web 界面上修改确认                  │
 └─────────────────────────────────────────────────────────────────┘
 ```
 
 ### 3.2 规则书知识库（RAG）详细设计
 
 #### 为什么需要
 
 CoC 规则书 300+ 页，你不可能把所有规则都硬编码到代码里，也不应该——太多边界情况、技能描述、道具效果、怪物数据。知识库让 AI 能在游戏过程中"查书"。
 
 #### 架构
 
 ```
 ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
 │  规则书 PDF  │    │  官方 FAQ    │    │  用户村规    │
 │  (CoC 7th)  │    │  (网页)      │    │  (用户上传)  │
 └──────┬───────┘    └──────┬───────┘    └──────┬───────┘
        │                   │                   │
        └───────────────────┼───────────────────┘
                            │
                            ▼
                   ┌──────────────────┐
                   │  文档分块 +      │
                   │  Embedding 索引  │
                   │  (Chunk 256-512) │
                   └──────────────────┘
                            │
                            ▼
                   ┌──────────────────┐
                   │  Vector DB       │
                   │  (SQLite + vec0  │
                   │   或 local 方案) │
                   └──────────────────┘
                            ▲
                            │
                   ┌──────────────────┐
                   │  运行时检索      │
                   │  (Retrieval)     │
                   │  每次 AI 调用时  │
                   │  自动 attach     │
                   │  相关规则段落    │
                   └──────────────────┘
 ```
 
 #### 什么时候触发检索
 
 | 场景 | 检索什么 | 用途 |
 |---|---|---|
 | 玩家问"极端成功的规则是什么" | 规则书→检定章节 | AI 直接回答玩家 |
 | AI 要描述某个法术的效果 | 规则书→法术章节 | AI 生成准确描述 |
 | 怪物特殊能力需要裁决 | 规则书→怪物数据 | AI / 代码共同参考 |
 | 玩家想做一个规则书没覆盖的动作 | 规则书→KP 裁决指南 | AI 给出合理裁定 |
 
 ### 3.3 模组自动解析器（Module Parser）
 
 这是你设想的"上传模组，AI 自动生成内容"这个功能的具体设计。
 
 #### 用户视角的流程
 
 ```
 1. 用户上传一个 CoC 模组 PDF（如 The Haunting 的扫图或文档）
 2. 系统显示: "正在解析模组，AI 正在提取场景和 NPC……"
 3. AI 解析完成 → 生成一个"可加载剧本包"（结构同剧本包格式）
 4. 用户预览生成结果: 场景列表、NPC 卡、线索链、怪物数据
 5. 用户可编辑/确认 → 确认后生成房间
 6. 房间创建后，AI KP 已了解全部模组内容
 ```
 
 #### 技术流程
 
 ```
 PDF/文档 ──→ 文本提取 ──→ AI 结构化解析 ──→ 人工预览/编辑 ──→ 剧本包 JSON
   │              │              │                       │
   │  PyMuPDF /   │  Markdown   │  LLM Call (GPT-4o)   │  场景/NPC/线索/
   │  pdfplumber  │  分段       │  带 Schema 的 Prompt  │  怪物全结构化了
   ▼              ▼              ▼                       ▼
 ```
 
 #### AI 解析 Prompt 结构示例
 
 ```
 System: 你是一个 CoC 模组解析器。你收到的是一份跑团模组的文本。
         请提取以下结构化信息:
         1. 场景列表（名称、进入条件、关键描述）
         2. NPC 列表（姓名、描述、性格、知道的信息）
         3. 线索链（每条线索的内容、在哪里发现、解锁什么）
         4. 怪物/敌对数据（名称、属性、特殊能力）
         5. 关键道具
         以 JSON 格式输出，严格遵循以下 Schema……
 ```
 
 **兜底策略**：如果 AI 解析失败，用户可以通过 Web 界面手动输入场景和 NPC 信息（或使用预装剧本）。
 
 ### 3.4 "规则书知识库方案会更简单吗？"——我的真实回答
 
 直接说结论：**对 MVP 来说，不完全更简单，但对产品长期发展来说是更正确的方向。**
 
 #### MVP 阶段（前两个月）的实际情况
 
 | 你的设想 | 实际难度 | 建议 |
 |---|---|---|
 | "AI 从规则书里自己学规则" | 🔴 不可靠 | **规则执行层还是得硬编码**。你没法让 AI 决定"这个检定应该过什么技能、难度多少"，因为一个错误判定会导致整个剧情崩掉 |
 | "上传模组，AI 自动提取 NPC 和场景" | 🟡 可行但有误差 | 可以做，但需要加一个人工审核界面。AI 提取 80% 正确，剩下 20% 需要人修 |
 | "规则书当知识库，AI 按需查" | 🟢 直接可做 | 这个是最容易实现的。MVP 里可以先做这一层 |
 
 #### 我的建议：MVP 阶段的分步策略
 
 **第一步（先做）：规则书 RAG 知识库**
 - 把 CoC 7th 规则书切片 + 向量化
 - AI 在游戏运行时按需检索，回答玩家规则问题、参考怪物数据
 - 这是"浅层知识库"，实现简单，效果明显
 
 **第二步（并行做）：确定性规则引擎硬编码**
 - 技能检定、Sanity、战斗轮这些高频核心规则写死
 - 这是游戏能"玩起来"的前提，不能依赖 AI
 
 **第三步（第二个月做）：模组上传自动解析**
 - 先支持英文/中文 CoC 模组 PDF
 - 输出剧本包 JSON，加入人工预览编辑界面
 - 这一步做完才真正实现"上传即玩"
 
 #### 未来理想状态
 
 ```
 用户上传规则书 PDF → AI 提取规则骨架 → 半自动生成规则引擎配置 + 知识库
 用户上传模组 PDF   → AI 提取剧情骨架 → 生成可加载的剧本包
 用户点"开团"       → 所有配置就绪     → AI DM 开始主持
 ```
 
 但这个"上传规则书 → 自动生成规则引擎"是进阶目标，不是两个月内能实现的。

 ### 4.2 核心数据流（带 AI/确定性标注）
 
 ```
 1. [🔵 AI] Intent Parser
    "我想翻一下办公桌的抽屉"
    → { action: "search", target: "desk_drawer", method: "tactile" }
    (兜底: 关键词匹配 + 场景可交互列表)
 
 2. [🟢 确定性] Scene Controller
    检查 desk_drawer 是否在当前场景存在
    确定检定类型: Spot Hidden (普通难度)
    通知前端显示检定面板
 
 3. [🟢 确定性] Player 点"掷骰"按钮
    Rule Engine 计算: 技能 65 / 骰出 47 → 成功
    标记线索 diary_fragment_1 为"已发现"
 
 4. [🔵 AI] Narration Generator
    输入: 成功检定 + 场景上下文 + 线索内容
    输出: "抽屉夹层里有个泛黄的信封，上面写着'不要打开地下室'……"
    (兜底: "你在抽屉里找到了线索")
 
 5. [🟢 确定性] Event Log + 分发
    记录事件
    按 visibility 字段分发:
    - 公开消息 → 所有人
    - 线索内容 → 仅发现者
 ```

 ## 五、模块详细设计
 
 ### 5.1 Game State Machine（游戏状态机）
 
 与 v0.1 一致。一个游戏实例的生命周期：LOBBY → PLAYING → ENDED，PLAYING 下每个 Scene 内部有微型状态机（ENTRY → INVESTIGATE → EXIT）。
 
 ### 5.2 Rule Engine（规则引擎）
 
 **重要调整**：v0.2 明确规则引擎为**全确定性代码，零 AI 调用**。
 
 ```typescript
 interface RuleEngine {
   skillCheck(params: {
     playerId: string;
     skillName: string;       // "Spot_Hidden" | "Library_Use"
     difficulty: "normal" | "hard" | "extreme";
     bonusDice?: number;      // -2 ~ +2，奖惩骰
   }): {
     success: boolean;
     grade: "critical" | "hard" | "normal" | "fail" | "fumble";
     roll: number;
     target: number;
   };
 
   sanityCheck(params: {
     currentSan: number; maxSan: number;
     loss: number; type: "mythos" | "violence" | "helplessness";
   }): {
     actualLoss: number;
     temporaryInsanity: boolean;
     indefiniteInsanity: boolean;
     boutsOfMadness?: string;
   };
 
   combatRound(actions: CombatAction[]): CombatRoundResult;
   damageRoll(diceFormula: string): number;
 }
 ```
 
 ### 5.3 AI Engine（详细 Prompt 示例）
 
 #### ① Intent Parser — 玩家自然语言 → 结构化动作
 
 **调用时机**：每一条玩家消息
 **System Prompt**：
 
 > 你是一个 TRPG 动作解析器。玩家的输入是自然语言，请将其解析为结构化的游戏动作。
 > 当前场景: {{场景描述}}
 > 可交互对象: {{场景内对象列表}}
 > 可用动作类型: [talk, search, move, use, attack, investigate, observe]
 > 输出 JSON: { "action_type": string, "target": string, "method": string, "confidence": number }
 > 如果无法解析: { "action_type": "unknown", "raw": "玩家原始输入" }
 
 **兜底**：AI 超时/confidence < 0.5 → 回退到关键词匹配 + 可交互对象菜单
 
 #### ② Narration Generator — 旁白/氛围描写
 
 **调用时机**：场景进入、检定结束、战斗事件、场景转换
 **System Prompt**：
 
 > 你是 CoC 的 KP，请生成一段氛围感强烈的叙事文本。
 > 风格: 克苏鲁式模糊不安，多用感官描写，每段 50-150 字
 > 当前场景: {{场景名称 + 描述}}
 > 触发事件: {{检定结果 / 玩家行动}}
 > 已知信息: {{已发现的线索摘要}}
 > 当前氛围: {{平静/紧张/恐怖}}
 
 **兜底**：AI 超时 → 场景预设描述文本
 
 #### ③ NPC Chat — 角色扮演对话
 
 **调用时机**：玩家主动与 NPC 对话
 **System Prompt**：
 
 > 你扮演: {{NPC.name}}, {{NPC.age}}岁, {{NPC.occupation}}
 > 性格: {{NPC.personality}}, 情绪: {{NPC.mood}}
 > 知道: {{NPC.knowledge}}
 > 不知道: {{NPC.ignorance}} ← 这部分绝对不能泄露
 > 用角色的语气说话，1-3 句话
 
 **兜底**：NPC 卡预设对话树（3-5 条默认回应）
 
 #### ④ Rule Query — 规则书检索问答
 
 **调用时机**：玩家问规则问题
 **流程**：玩家提问 → RAG 检索规则书 → 原文段落 + 问题 → LLM 生成回答（带出处）
 
 #### ⑤ Module Parser — 模组自动解析
 
 **调用时机**：用户上传模组 PDF（创建房间前）
 **流程**：PDF 文本提取 → 一次性 LLM 调用（大上下文）→ 结构化剧本包 JSON → 人工审核
 
 ### 5.4 Room & Session Manager
 
 与 v0.1 一致。关键设计：房间不要求注册，6 位邀请码/二维码上车，Guest Token 断线重连。
 
 ### 5.5 Event Log
 
 与 v0.1 一致。每局游戏是一个不可变事件流，用途：实时推送 + 复盘生成 + 未来回滚。

 ## 八、MVP 不做清单
 
 | 不做 | 原因 |
 |---|---|
 | 自动从规则书生成规则引擎代码 | 技术难度大，两个月内不可靠 |
 | 模组解析无需人工审核 | AI 提取有误差，必须有人审 |
 | 多规则系统同时支持 | MVP 只做 CoC，一次做好一个 |
 | 地图/Token 渲染 | CoC 是叙事驱动，文字+图片够用 |
 | 语音/视频 | 两个月做不完也不必要 |
 | 用户账号系统 | Guest Token + 房间 ID 足够 |
 | 移动端 Native App | PWA/Web 先验证 |
 | 剧本编辑器 | 剧本用 JSON 手写或 AI 解析生成 |
 | 跨游戏框架抽象 | MVP 只做 CoC，接口留扩展点 |
